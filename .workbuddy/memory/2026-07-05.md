## AI 无法回答问题排查（00:40 - 00:55）

### 问题现象
前端对话报"下游服务不可用"（错误码 5002），AI 不输出任何内容。

### 根因 1：DashScope SDK 兼容性 bug（已修复）
`backend/ai_service/adapter/dashscope_adapter.py:96` 用 `getattr(getattr(response, "output", None), "embeddings", None)` 访问 embeddings，但 dashscope SDK 1.26.2 的 `response.output` 返回的是 **dict** 不是对象，导致 `getattr(dict, "embeddings")` 返回 None → 报"向量化服务返回空结果" → 文档入库失败 → 知识库为空 → AI 返回 5003。
**修复**：改为先判断 `isinstance(output, dict)` 再 `.get("embeddings")`，兼容 dict 和对象两种返回。修复后直接调 ai_service embedding 接口正常（0.5 秒返回，维度 1024）。

### 根因 2：httpx 连接池死连接（已修复，待重启验证）
`backend/common/http_client.py` 的 `httpx.AsyncClient` 默认开 keep-alive，连接池复用连接。当下游服务 reload 重启后（如 ai_service 因代码改动触发热重载），池里旧连接失效，gateway/knowledge_service/chat_service 复用死连接 → `httpx.RequestError`（err 为空）→ 报 5002 "下游服务不可用"。
**修复**：`http_client.py:create_client` 改为 `limits=httpx.Limits(max_keepalive_connections=0)` 禁用 keep-alive，每次请求新建连接。代价是轻微性能损失，但彻底避免死连接问题。

## 新建会话无响应排查（09:47 - 10:06）

### 问题现象
前端点击"新建会话"按钮无反应，请求 hang 死。chat_service 认证过的请求全部超时 30 秒，未认证的 401 请求正常返回。

### 根因：aiomysql 连接池泄漏（已修复）
`common/database.py` 的 `DB` 类有严重 bug：
- `DB.__aenter__`: 先 `pool.acquire()` 拿上下文管理器，再手动 `__aenter__()` 取连接 → 池标记"使用中"
- `DB.__aexit__`: 调用 `conn.close()` 关闭物理连接，但**从不调用 `pool.acquire().__aexit__()`** → 连接永远不释放回池
- 每次用 `DB()` "吞"一个连接，池 maxsize=10，约10次DB操作后池耗尽
- 新请求 `pool.acquire()` 永远等不到连接 → hang 死

### 验证
- MySQL `SHOW PROCESSLIST` 只有 2 个连接（物理连接已被 close() 关掉）
- 直接 httpx 发带 x-user-id 的请求 → ReadTimeout（走DB就hang）
- 直接 httpx 发不带 x-user-id 的请求 → 401 正常（不走DB）
- 刚启动时前几次请求正常（池还有空闲），~10次后开始hang

### 修复
- `DB.__aexit__` 改为调用 `self._pool_ctx.__aexit__()` 正确释放回池（而不是 `conn.close()`）
- 同时 `.env` 中 MySQL/Redis/服务地址从 `localhost` 改为 `127.0.0.1`（避免 Windows localhost 优先解析 IPv6 ::1）

## 删除文档导致 Knowledge Service 崩溃修复（17:09）

### 问题现象
前端点击删除文档按钮 → 报"下游服务不可用"
- 文档列表能正常加载（GET 正常）
- 直接 curl knowledge_service :8003 DELETE → **退出码 56（连接被重置/Recv failure）**
- 说明 knowledge_service 在处理 DELETE 请求时**进程崩溃/hang 死**

### 根因1：ChromaDB 跨线程并发锁冲突（第一轮修复）
1. ingest 已隔离到独立 daemon 线程 + 独立事件循环
2. 但 **ChromaDB PersistentClient 是全局单例 `_client`**，主线程和 ingest 线程共享
3. 当 ingest 线程正在 `add_chunks()` 写入向量时，主线程同时执行 `delete_by_document()` 删除
4. 两个线程**并发操作同一个 ChromaDB client 的同一 collection** → SQLite/WAL 文件锁冲突 → 进程 hang/crash

### 第一轮修复（ChromaDB 锁 + 超时）
- `vector_store.py`: 新增 `threading.Lock()` 全局锁，所有 ChromaDB 操作串行化
- `document_service.py` / `kb_service.py`: ChromaDB 操作外包 30s 超时保护
- `delete_by_document()`: collection 不存在时容错跳过

### 根因2（17:17 截图新错误）：aiomysql "attached to a different loop"
截图显示红色错误：
```
Task got a Future <Pool.wakeup()> attached to a different loop
```

**根因**：`database.py` 的 `_pool` 是**全局单例变量**：
1. 主线程启动 → `main.py:lifespan` 调用 `create_pool()` → 创建 pool_A 绑定**主事件循环**
2. ingest 线程启动 → `_ingest_thread_main()` 也调用 `create_pool()` → **覆盖**全局 `_pool` 为 pool_B 绑定 **ingest 事件循环**
3. 主线程处理请求时调用 `DB()` → `get_pool()` 返回 **pool_B**（绑定到 ingest 循环）
4. 主循环尝试 acquire pool_B → **"attached to a different loop" 崩溃**

### 最终修复（ingest 回退到主事件循环 v3）
**核心决策**：放弃独立线程方案，回退到 `asyncio.create_task()`

原因：独立线程无法安全共享 database.py 全局连接池。要彻底解决需要重写 database.py 支持多连接池——代价太高。

防护措施（确保不拖死主服务）：
1. ✅ ChromaDB 操作通过 `asyncio.to_thread()` 放线程池 + **chroma_lock 串行化**
2. ✅ HTTP 调用(ai_client)使用**独立 httpx client** + 60s 超时
3. ✅ 整体入库流程 **wait_for(120s) 超时保护**

### 修改的文件

## 删除文档永久 hang 修复（17:23）

### 问题现象
curl -v 测试 DELETE 请求 → **近 2 分钟无响应**（hang 死），连 GET 请求也无法响应
- knowledge_service 进程存在（PID 15728，端口 8003 监听中）
- 但所有 HTTP 请求都不返回 → **整个服务 hang**

### 根因：chroma_lock 无超时 + 对 pending 文档执行无效 ChromaDB 操作

两层问题叠加：

1. **`threading.Lock()` 无超时**：
   - `with chroma_lock:` 会**无限等待**
   - 当 ingest 的 `add_chunks()` 持锁后 ChromaDB 内部操作 hang（Windows SQLite/WAL 锁）
   - `delete_by_document` 等锁 → **永久阻塞**

2. **对所有状态文档都执行 ChromaDB 删除**：
   - 截图中文档状态全是"处理中"(pending)，从未成功写入过向量
   - 但代码对 pending/failed 文档也调用 `delete_by_document()`
   - → 尝试获取 chroma_lock → 等 ingest 释放 → ingest 在 ChromaDB hang → **永久等锁**
   - 更糟糕的是，即使没有 ingest 在运行，对不存在的 collection 做 `get_collection()` + `delete()` 也可能触发 ChromaDB 内部异常/hang

### 修复（3 层防护）

#### 层1：chroma_lock 改为带超时的 acquire（vector_store.py）
- 新增 `_acquire_lock()` 函数：`lock.acquire(timeout=30)` 而非 `with lock`
- 所有 4 个函数(create_collection/add_chunks/delete_by_document/delete_collection) 都改用 try/finally + acquire/release
- 超时抛 RuntimeError，调用方 catch 后跳过

#### 层2：仅对 ready 状态文档清理 ChromaDB（document_service.py）
```python
if status == "ready":
    # 只有成功入库的文档才有向量数据需要清理
    await asyncio.to_thread(delete_by_document, ...)
else:
    # pending/failed 文档从未写过向量，直接跳过
    logger.info("跳过 ChromaDB 向量清理...")
```
这是**最关键的改动**——截图中所有文档都是"处理中"状态，删除时根本不需要碰 ChromaDB

#### 层3：ChromaDB 操作外包 asyncio.wait_for(30s) 保持不变
- 即使 chroma_lock 获取成功，ChromaDB 内部操作本身也可能 hang
- 30 秒超时确保不拖死主服务

### 修改的文件
- **`vector_store.py`** — `with chroma_lock:` 全部改为 `acquire(timeout=30)` + try/finally
- **`document_service.py`** — 仅 `status=="ready"` 时才调 delete_by_document
