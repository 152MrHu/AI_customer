## Project Overview

AI-powered intelligent customer service system (AI 智能客服系统) using RAG (Retrieval-Augmented Generation) with SSE streaming. Backend is Python/FastAPI microservices; frontend is React 18 + Ant Design 5 + Vite. LLM backend is Alibaba DashScope (Tongyi Qianwen / Qwen).

## Commands

### Backend (Python)

```bash
# Install dependencies
cd backend && pip install -r requirements.txt

# Initialize database (first time only)
python scripts/init_db.py      # Creates database + tables from sql/init.sql
python scripts/init_admin.py   # Seeds admin/admin123 and testuser/user123

# Start each service in its own terminal (order doesn't matter):
python -m user_service.main        # Port 8001
python -m chat_service.main        # Port 8002
python -m knowledge_service.main   # Port 8003
python -m ai_service.main          # Port 8004
python -m gateway.main             # Port 8000 (start last — users hit this)

# All services must be running for full functionality.
```

### Frontend (React)

```bash
cd frontend && npm install
npm run dev        # Dev server on :5173, proxies /api → localhost:8000
npm run build      # Production build → frontend/dist/
```

### Configuration

Copy `.env.example` → `backend/.env` and fill in `MYSQL_PASSWORD`, `DASHSCOPE_API_KEY`, and `JWT_SECRET_KEY`. The config is read by `backend/common/config.py` (pydantic-settings, reads from `backend/.env` with absolute-path resolution so services can be started from any directory).

## Recent Changes (2026-07-06)

### Security Hardening
- **Service binding**: Internal services (user/chat/knowledge/ai) now bind to `127.0.0.1` — only gateway is on `0.0.0.0`
- **Rate limiting**: Extended to login, register, and document upload endpoints. Uses IP-based limiting for public endpoints, user-based for authenticated ones. Returns HTTP 429.
- **Auth responses**: JWT middleware now returns HTTP 401 (was 200) for auth failures; rate limit returns 429.
- **Error messages**: Generic exception handler no longer leaks `str(exc)` to clients; logs full traceback internally.
- **JWT `jti` claim**: Each token now includes a unique `jti` (UUID4 hex) for per-token revocation tracking.
- **Filename sanitization**: Uploaded filenames are sanitized (path traversal chars removed, restricted charset).
- **Magic bytes validation**: PDF and DOCX files are validated by content magic bytes, not just extension.
- **Prompt injection protection**: `sanitize_query()` in `ai_service/retrieval/prompt.py` detects known injection patterns and strips control characters.
- **Login lockout**: Now tracks by `user_id` instead of account string, preventing username/phone alternation bypass.
- **`SELECT *` removed**: User queries use explicit column lists (no `password_hash` leak risk).

### New Features
- **User profile management**: `PUT /api/user/profile` (edit) + `PUT /api/user/password` (change password)
- **Password reset**: `POST /api/user/reset-password` (admin) + `POST /api/user/forgot-password` (self-service via username+phone)
- **Message feedback**: `POST /api/chat/messages/{id}/feedback` — thumbs up/down with optional comment. New `message_feedback` table.
- **Human handoff**: `POST /api/chat/sessions/{id}/handoff` — create ticket; admin claim/resolve flow. New `handoff_tickets` table.

### Code Quality
- **Shared SSE module**: `common/sse.py` replaces duplicate copies in `chat_service/clients/sse.py` and `ai_service/utils/sse.py` (marked deprecated).
- **Dead code removed**: `chroma_worker.py` deleted, `chromadb` removed from `requirements.txt`.
- **Request tracing**: `X-Request-Id` header generated by gateway, logged and forwarded to downstream services.
- **Health checks**: All 5 services now have `GET /health` endpoints.

## Architecture

### 5 Microservices + Gateway

All backend services are independent FastAPI apps. The **gateway** (:8000) is the only public entry point — it authenticates via JWT, enforces rate limits, logs requests, then reverse-proxies to downstream services based on URL prefix:

| Prefix | Service | Port |
|--------|---------|------|
| `/api/user` | user_service | 8001 |
| `/api/chat` | chat_service | 8002 |
| `/api/knowledge` | knowledge_service | 8003 |
| `/api/ai` | ai_service | 8004 |

After JWT validation, the gateway injects `X-User-Id` and `X-User-Role` headers. Downstream services read these via the shared `get_current_user()` dependency in `backend/common/dependencies.py`. Public endpoints (login, register) are whitelisted in `gateway/routes.py`.

### SSE Chat Flow (the critical path)

1. Browser → gateway → **chat_service** (:8002) — user sends a message
2. chat_service → **ai_service** (:8004) — requests RAG streaming response
3. ai_service → **knowledge_service** (:8003) — semantic vector search
4. ai_service streams LLM tokens back through the chain as SSE frames
5. chat_service saves the complete message to MySQL when streaming finishes

### Dual Chat Modes

- **`kb` mode**: Strict RAG — only answers from uploaded knowledge base documents. If no relevant chunks found (similarity < threshold), refuses to answer.
- **`assistant` mode**: General assistant with DashScope web search enhancement (`enable_search`).

### Vector Store: SQLite + NumPy (not ChromaDB)

Originally used ChromaDB but migrated to a custom lightweight SQLite + numpy vector store (`knowledge_service/vector_store.py`) to avoid segfaults on Windows/Python 3.13. The ChromaDB dependency is still in `requirements.txt` and `chroma_worker.py` exists but is not used. Vector DB file: `chroma_data/vec_store.db`.

The `ai_service` never accesses the vector store directly — it calls `knowledge_service` via HTTP (`/api/knowledge/search`, `/api/knowledge/count`) to avoid SQLite file-lock conflicts across processes.

### Async Document Ingestion

When a document is uploaded to `knowledge_service`, it spawns an `asyncio.create_task()` that: parses → chunks (size=600, overlap=100) → calls `ai_service` for embeddings → writes to SQLite vector store. Timeout: 120s. On startup, `knowledge_service` auto-retries any documents stuck in `pending`/`failed` status.

### Shared Backend Module (`backend/common/`)

All services import from `backend/common/`, which provides:
- `config.py` — `Settings` singleton from env vars
- `database.py` — aiomysql connection pool + `@asynccontextmanager` DB helper
- `redis_client.py` — Redis async client (token blacklist on logout, rate-limit counters, session cache)
- `jwt_utils.py` — JWT create/decode
- `security.py` — bcrypt password hashing
- `response.py` — Unified `{code, message, data}` response format
- `exceptions.py` — `BusinessError` with error codes
- `dependencies.py` — `get_current_user()` / `require_admin()` FastAPI dependencies (read `X-User-Id`/`X-User-Role` headers)
- `http_client.py` — Shared httpx async client for inter-service HTTP calls
- `pagination.py` — `PageParams` dependency (page/page_size)

### Frontend State

- **Auth**: `AuthContext` provides token/user/role. JWT stored in localStorage. `request.js` auto-attaches `Authorization: Bearer` header and redirects to `/login` on 401.
- **SSE client**: `frontend/src/api/sse.js` — custom SSE reader using `fetch` + `ReadableStream` for streaming chat responses.
- **Routes**: `/login`, `/register`, `/chat` (protected), `/admin/*` (admin-only, protected).

### Database Tables (MySQL)

`users`, `knowledge_bases`, `documents`, `sessions`, `messages` — DDL in `backend/sql/init.sql`.

### Key Design Decisions

- Session limit: 100 per user, auto-deletes oldest when exceeded
- Multi-turn context: last 20 messages sent to LLM
- Session auto-titling: first user message becomes session title
- Rate limiting: 10 req/min per user on chat message endpoint (gateway + Redis)
- JWT logout: token blacklisted in Redis with TTL matching remaining validity
- Default accounts: `admin`/`admin123` (admin), `testuser`/`user123` (user)
