# 调研报告

## 1. 项目背景与选题意义

### 1.1 项目背景

随着企业数字化转型的深入推进，客户服务作为企业与用户之间最重要的触点之一，正经历着从传统人工客服向智能化、自动化方向的深刻变革。传统的客服模式面临以下突出问题：

- **人力成本高**：需要大量客服人员 7×24 小时轮班值守
- **响应效率低**：高峰期用户等待时间长，体验差
- **知识传递难**：客服培训周期长，专业知识难以全面覆盖
- **重复性问题占比高**：据统计，80%以上的用户咨询属于常见问题，完全可以自动化处理

与此同时，大语言模型（LLM）技术的成熟为智能客服带来了全新的可能性。基于 RAG（检索增强生成）技术，系统能够结合企业自有知识库，提供准确、可靠、可溯源的自动化问答服务，有效解决了传统 LLM "幻觉"问题和知识更新滞后的问题。

### 1.2 选题意义

本项目"AI 智能客服系统"的选题意义体现在以下几个方面：

**（1）技术实践价值**

项目采用微服务架构 + RAG + SSE 流式输出的技术方案，涵盖前后端分离、服务拆分、网关路由、向量检索、Prompt 工程等核心技术点，是"微服务 + AI"工程化实践的理想载体。

**（2）业务应用价值**

系统支持知识库文档上传、智能问答、人工转接、消息反馈等完整客服流程，可直接应用于校园办事咨询、企业制度问答、电商售前售后等多种场景，具有较强的通用性和实用性。

**（3）教学示范价值**

项目遵循"业务与 AI 解耦"的设计原则，AI 能力作为独立服务存在，业务服务不直接依赖模型厂商 API。这种架构设计体现了工程化 AI 应用的最佳实践，适合作为教学案例。

### 2.3 差距分析

当前市场上的智能客服产品主要存在以下不足：

1. **模型厂商锁定**：大多数产品与特定模型厂商深度绑定，切换成本高
2. **业务与 AI 耦合**：业务逻辑与 AI 调用深度耦合，不利于维护和演进
3. **知识更新延迟**：部分产品需要人工维护 FAQ，知识更新周期长
4. **数据隐私顾虑**：SaaS 模式下企业数据存储在第三方平台
5. **定制化程度低**：中小企业无法根据自身业务特点深度定制

本项目的设计思路正是针对上述痛点：通过适配器模式实现模型可替换，通过微服务架构实现业务与 AI 解耦，通过 RAG 知识库实现文档级知识自动入库。

## 3. 核心业务场景分析

### 3.1 知识库智能问答场景

**场景描述**：用户向系统提出业务咨询问题，系统基于上传的知识库文档进行语义检索，找到最相关的文档片段，结合 LLM 生成准确、可溯源的答案。

**典型流程**：
1. 管理员上传企业制度、产品手册、FAQ 等文档
2. 系统自动解析文档、切块、向量化、入库
3. 用户输入问题
4. 系统进行语义检索，找到相关文档片段
5. LLM 基于检索结果生成回答，并标注引用来源

**关键指标**：
- 检索相似度阈值：0.5
- 单次检索返回 Top-5 相关片段
- 文档切块大小：600 字符，重叠 100 字符

### 3.2 通用智能助手场景

**场景描述**：用户开启"通用助手"模式，系统不依赖知识库内容，而是基于 LLM 自身能力并启用联网搜索，提供更广泛的问答服务。

**典型流程**：
1. 用户切换到"通用助手"模式
2. 输入任意问题
3. 系统调用 DashScope 联网搜索增强功能
4. LLM 综合搜索结果和自身知识生成回答

### 3.3 人工客服转接场景

**场景描述**：当 AI 无法满足用户需求时（如复杂投诉、特殊业务办理），用户可请求转接人工客服。

**典型流程**：
1. 用户在对话中点击"转人工"
2. 系统创建转接工单
3. 客服人员在工单列表中认领工单
4. 客服进入会话，与用户实时沟通
5. 问题解决后客服关闭工单

### 3.4 知识库管理场景

**场景描述**：管理员对知识库和文档进行全生命周期管理。

**典型流程**：
1. 管理员创建知识库
2. 上传 PDF/DOCX/TXT/MD/CSV 格式文档
3. 系统自动入库（解析→切块→向量化→存储）
4. 管理员可查看文档列表和状态
5. 支持删除文档（同步删除向量数据）

## 4. 技术路线初步分析

### 4.1 后端技术栈

| 技术 | 版本 | 选型理由 |
|------|------|----------|
| Python | 3.12+ | AI/ML 生态丰富，开发效率高 |
| FastAPI | ≥0.110 | 高性能异步 Web 框架，原生支持 SSE 流式 |
| Uvicorn | ≥0.27 | 轻量级 ASGI 服务器 |
| MySQL | 8.0+ | 成熟的关系型数据库，事务支持完善 |
| Redis | 7.0+ | 高性能缓存，支撑 Token 黑名单和限流 |
| aiomysql | ≥0.2 | 异步 MySQL 驱动，适合 FastAPI |
| bcrypt | ≥4.1 | 安全的密码哈希算法 |
| PyJWT | ≥2.8 | JWT 签发与验证 |
| httpx | ≥0.27 | 异步 HTTP 客户端，服务间调用 |

### 4.2 前端技术栈

| 技术 | 版本 | 选型理由 |
|------|------|----------|
| React | 18.x | 主流前端框架，生态丰富 |
| Ant Design | 5.x | 企业级 UI 组件库，中文支持好 |
| Vite | 5.x | 新一代前端构建工具，开发体验优秀 |
| React Router | 6.x | 声明式路由管理 |

### 4.3 AI 技术栈

| 技术 | 选型理由 |
|------|----------|
| 阿里云 DashScope（通义千问） | 国内可访问，API 稳定，价格合理 |
| Qwen-Turbo（LLM） | 性价比高，适合对话场景 |
| text-embedding-v3（向量化） | 1024 维向量，检索效果好 |
| RAG（检索增强生成） | 结合知识库，减少幻觉，提高准确性 |
| SSE（Server-Sent Events） | 实现打字机效果的流式输出 |

### 4.4 向量存储方案

项目最初使用 ChromaDB 作为向量存储，但在 Windows + Python 3.13 环境下遇到 ChromaDB Rust 后端的段错误问题。经过技术调研后，迁移到自研的轻量级向量存储方案：

- **存储层**：SQLite（向量以 JSON 序列化存储）
- **检索层**：NumPy 余弦相似度计算
- **优势**：纯 Python 实现，无 Rust/C++ 依赖，兼容性好，零部署成本

## 5. AI 能力可落地场景分析

### 5.1 已实现的 AI 能力

| AI 能力 | 实现方式 | 应用场景 |
|---------|----------|----------|
| **RAG 智能问答** | 向量检索 + LLM 生成 + SSE 流式 | 基于知识库的精准问答 |

| **消息反馈** | 用户赞/踩评分 | AI 回答质量持续改进 |

## 6. 本组项目定位与目标

### 6.1 项目定位

本项目定位为**面向中小企业的轻量级 AI 智能客服系统**，核心特色：

- **微服务架构**：5 个独立微服务 + API 网关，体现工程化架构设计
- **AI 服务化**：AI 能力作为独立服务，与业务逻辑解耦，模型可替换
- **RAG 知识库**：支持多格式文档上传，自动入库，精准问答
- **双模式对话**：知识库模式（严格 RAG）+ 通用助手模式（联网搜索）
- **人工协同**：支持人工客服转接，AI 与人工协同工作

### 6.2 目标用户

| 用户角色 | 核心需求 | 功能范围 |
|----------|----------|----------|
| **普通用户** | 智能问答、自助查询 | 会话管理、消息发送、反馈评价、转人工 |
| **客服人员** | 处理复杂问题、查看工单 | 工单认领、会话接管、在线回复 |
| **管理员** | 维护知识库、管理系统 | 知识库管理、文档管理、用户管理 |

### 6.3 项目目标

**功能目标**：
- 实现基于知识库的 RAG 智能问答
- 支持多格式文档上传与自动入库
- 实现 SSE 流式对话输出
- 支持多轮对话记忆
- 实现人工客服转接流程
- 提供消息反馈机制

**技术目标**：
- 搭建 5 个微服务 + API 网关的完整架构
- 实现 LLM 适配器模式，支持模型切换
- 实现 Prompt 注入防护等安全措施
- 前后端分离，接口标准化

**质量目标**：
- 知识库模式下回答基于文档内容，不编造信息
- 检索相似度低于阈值时明确告知用户
- AI 服务不可用时优雅降级
