# 概要设计说明书

## 1. 系统总体架构

### 1.1 架构概述

本系统采用**微服务架构**，将系统拆分为 5 个独立服务 + 1 个 API 网关。每个服务独立部署、独立运行，通过 HTTP/SSE 协议通信。AI 能力作为独立服务存在，与业务服务逻辑解耦、物理隔离。

### 1.3 架构层次说明

| 层次 | 组件 | 职责 |
|------|------|------|
| **交互层** | React 前端 | 用户界面、SSE 流式接收、状态管理 |
| **接入与控制层** | API Gateway | 统一入口、CORS、JWT 鉴权、限流、日志、代理转发 |
| **业务能力层** | User Service | 注册、登录、用户管理、权限控制 |
|  | Chat Service | 会话管理、消息流式、反馈、转接工单 |
|  | Knowledge Service | 知识库管理、文档上传入库、向量检索 |
|  | AI Service | RAG 问答、文本向量化、LLM 适配 |
| **数据与集成层** | MySQL | 用户、会话、消息、知识库等业务数据 |
|  | Redis | Token 黑名单、限流计数、缓存 |
|  | SQLite + NumPy | 向量数据的存储与检索 |
|  | DashScope | 大语言模型（Qwen）+ 文本嵌入模型 |
| **运维与治理层** | 日志 | 统一日志格式，控制台 + 文件输出 |
|  | 健康检查 | 所有服务 /health 端点 |
|  | X-Request-Id | 请求追踪标识透传 |

## 3. 微服务拆分说明

### 3.1 服务边界

| 服务 | 绑定地址 | 负责 | 不负责 |
|------|----------|------|--------|
| **Gateway** | 0.0.0.0:8000 | CORS、JWT鉴权、限流、请求日志、反向代理 | 业务逻辑、数据存储 |
| **User Service** | 127.0.0.1:8001 | 注册、登录、个人信息、密码、用户管理 | 会话管理、知识库、AI |
| **Chat Service** | 127.0.0.1:8002 | 会话CRUD、消息流式、反馈、转接工单 | 用户认证、LLM调用、向量检索 |
| **Knowledge Service** | 127.0.0.1:8003 | 知识库CRUD、文档管理、向量存储与检索 | 用户管理、对话管理、LLM调用 |
| **AI Service** | 127.0.0.1:8004 | LLM对话、文本向量化、RAG流程编排、Prompt管理 | 用户管理、会话存储、文档管理 |

### 3.2 服务拆分原则

1. **按能力域拆分**：身份域（User）、对话域（Chat）、知识域（Knowledge）、AI 域（AI）
2. **每个服务独立数据**：各服务有自己的数据库表和业务逻辑
3. **AI 与业务解耦**：AI Service 是唯一直接调用模型厂商的服务
4. **内部服务隔离**：仅网关对外暴露，其他服务绑定 127.0.0.1
5. **可独立部署**：每个服务独立启动/停止，互不影响

### 3.3 能力域对应关系

| 能力域 | 对应服务 | 核心能力 |
|--------|----------|----------|
| **身份域** | User Service | 注册、登录、认证、权限、角色管理 |
| **核心业务域** | Chat Service | 会话管理、消息管理、流式对话 |
| **流程域** | Chat Service（Handoff） | 转接工单创建→认领→解决→关闭 |
| **AI 能力域** | AI Service | LLM对话、向量化、RAG、Prompt管理 |
| **内容/资源域** | Knowledge Service | 知识库、文档管理、向量存储、解析入库 |
| **公共支撑域** | common/ | 配置、数据库、Redis、JWT、安全、日志 |

## 5. 服务调用关系

### 5.1 服务间调用矩阵

| 调用方 | 被调用方 | 协议 | 用途 |
|--------|----------|------|------|
| Browser | Gateway | HTTP/SSE | 所有客户端请求入口 |
| Gateway | All Services | HTTP/SSE | 反向代理转发 |
| Chat Service | AI Service | HTTP/SSE | 发送 RAG 问答请求，接收 SSE 流 |
| Knowledge Service | AI Service | HTTP | 文档向量化（调用 embedding） |
| AI Service | Knowledge Service | HTTP | 向量检索 + 计数查询 |
| Chat Service | AI Service | HTTP | 客服消息发送（agent-message） |

### 5.2 调用约束

1. **业务服务不直连模型厂商**：Chat Service / Knowledge Service 不直接调用 DashScope，统一通过 AI Service
2. **AI Service 不直连向量库**：通过 HTTP 调用 Knowledge Service 的向量接口，避免跨进程 SQLite 文件锁
3. **跨服务写数据禁止**：每个服务的数据只能由其自身修改，其他服务通过接口获取

### 5.3 请求追踪

网关注入 `X-Request-Id`（UUID），通过 `common/http_client.py` 自动透传到下游服务的请求头中，并记录在日志中，实现全链路追踪。

## 6. AI 服务在整体架构中的位置

```
                   ┌──────────────────────────┐
                   │      业务服务层            │
                   │  User / Chat / Knowledge  │
                   └────────────┬─────────────┘
                                │ HTTP API 调用
                                │ (不直接依赖模型厂商)
                                ▼
                   ┌──────────────────────────┐
                   │      AI Service          │
                   │  ┌────────────────────┐  │
                   │  │   LLM Adapter      │  │
                   │  │  (抽象基类 + 工厂)  │  │
                   │  ├────────────────────┤  │
                   │  │ DashScopeAdapter   │  │
                   │  │ (通义千问 / Qwen)   │  │
                   │  ├────────────────────┤  │
                   │  │ [预留] OpenAIAdapter│  │
                   │  └────────────────────┘  │
                   │  ┌────────────────────┐  │
                   │  │   RAG Service      │  │
                   │  │ Prompt / 注入防护   │  │
                   │  │ 检索编排 / 流式生成 │  │
                   │  └────────────────────┘  │
                   └────────────┬─────────────┘
                                │ API 调用
                                ▼
                   ┌──────────────────────────┐
                   │     模型厂商层             │
                   │   阿里云 DashScope         │
                   │  (Qwen-Turbo / Qwen-Plus) │
                   └──────────────────────────┘
```

**关键设计原则**：
- AI Service 是唯一知道"用哪个模型"的服务
- 业务层只依赖 AI Service 接口，不依赖模型厂商
- 切换模型只需修改 adapter 层配置，业务代码零改动
- 适配器抽象基类定义了标准的 `chat_stream()` 和 `embed()` 接口

## 7. 关键技术选型说明

### 7.1 框架与运行时

| 技术 | 选型理由 |
|------|----------|
| **FastAPI** | 原生异步、SSE 支持、自动 API 文档、Pydantic 验证 |
| **Uvicorn** | 轻量级 ASGI 服务器，适合微服务 |
| **React 18 + Ant Design 5** | 生态系统成熟，企业级组件库，中文友好 |
| **Vite** | 极速热更新，ESBuild 预构建 |

### 7.2 数据存储

| 技术 | 选型理由 |
|------|----------|
| **MySQL 8.0** | 成熟稳定，事务支持，aiomysql 异步驱动 |
| **Redis 7.0** | 高性能缓存，天然支持过期策略（Token 黑名单/限流） |
| **SQLite + NumPy** | 替代 ChromaDB，纯 Python 无 segfault 风险，零部署成本 |

### 7.3 AI 与模型

| 技术 | 选型理由 |
|------|----------|
| **DashScope (通义千问)** | 国内可访问，API 稳定，性价比高 |
| **Qwen-Turbo** | 适合对话场景，速度快，成本低 |
| **Qwen-Plus** | 支持联网搜索增强（enable_search） |
| **text-embedding-v3** | 1024 维向量，语义检索效果好 |
| **适配器模式** | 模型可替换，预留 OpenAI 扩展位 |

### 7.4 安全

| 技术 | 选型理由 |
|------|----------|
| **JWT + jti** | 无状态认证，jti 支持单 token 撤销 |
| **bcrypt (rounds=12)** | 业界标准的密码哈希算法 |
| **Redis 黑名单** | 高性能 token 撤销机制 |
| **正则注入检测** | 轻量级 Prompt 注入防护 |
| **Magic bytes 校验** | 文件类型安全验证 |

### 7.5 为什么不选某些技术

| 未选技术 | 原因 |
|----------|------|
| **Nacos / Sentinel** | 4 人小组项目规模不需要注册中心和服务治理，网关直连即可 |
| **Docker / K8s** | 本地开发直接运行更快捷，生产部署可后续容器化 |
| **ChromaDB** | Windows + Python 3.13 段错误，已迁移到 SQLite + NumPy |
| **Elasticsearch** | 部署重，SQLite + NumPy 足够支撑课程项目规模 |
| **消息队列 (RabbitMQ/Kafka)** | 当前异步任务（文档入库）用 asyncio.create_task 即可，无需外部队列 |
| **Spring Boot / Java** | 团队偏好 Python 快速开发，AI 生态更好 |
