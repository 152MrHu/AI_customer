## 2. 数据库总体设计

### 2.1 数据库概述

系统使用 MySQL 8.0 作为主数据库，存储用户、知识库、文档、会话、消息、反馈、工单等业务数据。同时使用 SQLite 存储向量数据（文档切块的 Embedding 向量），使用 Redis 作为缓存层（Token 黑名单、限流计数、会话缓存）。

### 2.2 数据存储分层

```
┌──────────────────────────────────────────┐
│              Redis (缓存层)               │
│  Token黑名单 / 限流计数 / 登录锁 / 会话缓存  │
├──────────────────────────────────────────┤
│           MySQL 8.0 (业务数据层)           │
│  users / knowledge_bases / documents      │
│  sessions / messages / feedback / tickets │
├──────────────────────────────────────────┤
│        SQLite (向量数据层)                 │
│  vec_chunks (embedding 向量存储)           │
└──────────────────────────────────────────┘
```

### 2.3 数据库访问方式

| 服务 | 数据源 | 访问方式 |
|------|--------|----------|
| User Service | MySQL | aiomysql 连接池，通过 common/database.py |
| Chat Service | MySQL | aiomysql 连接池，通过 common/database.py |
| Knowledge Service | MySQL + SQLite | aiomysql + sqlite3 |
| AI Service | 无直接数据库 | 通过 HTTP 调用 Knowledge Service |

### 2.4 命名约定

- **表名**：小写 + 下划线，使用复数或集合名（如 `users`, `sessions`）
- **主键**：`{表名单数}_id`，如 `user_id`, `session_id`
- **外键**：与被引用表主键同名
- **索引**：`idx_{表名}_{字段名}` 或 `uk_{表名}`（唯一索引）
- **时间字段**：`created_at`, `updated_at` 使用 DATETIME 类型

```sql
    user_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL,

    password_hash VARCHAR(255) NOT NULL,
    role VARCHAR(20) NOT NULL DEFAULT 'user',
    status TINYINT NOT NULL DEFAULT 1,

    UNIQUE KEY uk_username (username),
    UNIQUE KEY uk_phone (phone),
    KEY idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户信息表';
```

## 5. 缓存数据设计说明

### 5.1 Redis 数据结构

| Key 模式 | 类型 | 用途 | TTL |
|----------|------|------|:---:|
| `blacklist:{jti}` | String | Token 黑名单 | Token 剩余有效期 |
| `rate_limit:login:{ip}` | String(计数器) | 登录限流 | 60s |
| `rate_limit:register:{ip}` | String(计数器) | 注册限流 | 60s |
| `rate_limit:ai:{user_id}` | String(计数器) | AI 问答限流 | 60s |
| `login_fail:{user_id}` | String(计数器) | 登录失败计数 | 1800s（30min）|
| `reset_pwd:{user_id}` | String(计数器) | 密码重置限流 | 3600s |
| `session_cache:{session_id}` | Hash | 会话缓存（预留） | 按需 |

### 5.2 Token 黑名单机制

```python
# 用户登出时
await redis.set(f"blacklist:{jti}", "1", ex=token_remaining_seconds)

# JWT 鉴权时
if await redis.exists(f"blacklist:{jti}"):
    raise HTTPException(401, "Token已失效")
```

### 5.3 限流实现

```python
# Redis 原子计数器 + 过期时间
key = f"rate_limit:ai:{user_id}"
count = await redis.incr(key)
if count == 1:
    await redis.expire(key, 60)
if count > 10:
    raise HTTPException(429, "请求过于频繁，请稍后重试")
```

## 6. 向量数据设计说明

### 6.1 概述

系统使用独立的 SQLite 数据库存储文档切块的 Embedding 向量。这是自研的轻量级向量存储方案，替代了 ChromaDB，消除了 Rust/C++ 依赖带来的兼容性问题。

### 6.2 存储位置

```
chroma_data/vec_store.db
```

### 6.3 表结构

| 序号 | 字段名 | 数据类型 | 说明 |
|:---:|--------|----------|------|
| 1 | kb_id | INTEGER | 知识库 ID（复合主键） |
| 2 | document_id | INTEGER | 文档 ID（复合主键） |
| 3 | chunk_index | INTEGER | 切块索引（复合主键） |
| 4 | file_name | TEXT | 源文件名 |
| 5 | chunk_text | TEXT | 切块文本内容 |
| 6 | embedding | TEXT | JSON 序列化的向量（1024 维 float 列表） |

**主键**：(kb_id, document_id, chunk_index) 联合主键

### 6.4 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS vec_chunks (
    kb_id INTEGER NOT NULL,
    document_id INTEGER NOT NULL,
    chunk_index INTEGER NOT NULL,
    file_name TEXT NOT NULL,
    chunk_text TEXT NOT NULL,
    embedding TEXT NOT NULL,
    PRIMARY KEY (kb_id, document_id, chunk_index)
);
```

### 6.5 向量检索算法

使用 NumPy 进行余弦相似度计算：

$$\text{cosine\_similarity}(A, B) = \frac{A \cdot B}{\|A\| \cdot \|B\|}$$


### 6.6 向量维度与模型

| 模型 | 维度 | 用途 |
|------|:---:|------|
| text-embedding-v3 (DashScope) | 1024 | 文本向量化 |
| 相似度计算方式 | — | 余弦相似度 |
| 存储格式 | — | JSON 序列化后 TEXT 存储 |

### 6.7 数据生命周期

```
文档上传 → 解析 → 切块 → 向量化 → 写入 vec_chunks
文档删除 → 从 vec_chunks 中删除 (DELETE WHERE kb_id=? AND document_id=?)
知识库删除 → 清空该 kb_id 所有记录
```

### 6.8 性能特征

| 指标 | 估算值 |
|------|:---:|
| 单向量存储大小 | ~8KB（1024 × 4 bytes float32 + JSON 开销） |
| 百篇文档向量容量 | ~50MB |
| 千条向量检索耗时 | < 100ms |
| 可支撑知识库文档数 | 1000+ 篇（课程项目规模） |

## 7. 数据规范与约定

### 7.1 字符集

- 所有 MySQL 表统一使用 `utf8mb4` 字符集和 `utf8mb4_unicode_ci` 排序规则
- 支持中文、英文、emoji 等全部 Unicode 字符

### 7.2 时间字段

- 统一使用 DATETIME 类型（非 TIMESTAMP），避免 2038 问题
- 默认值使用 `CURRENT_TIMESTAMP`
- 更新时间使用 `ON UPDATE CURRENT_TIMESTAMP`

### 7.3 ID 生成策略

- 所有主键使用 BIGINT AUTO_INCREMENT
- JWT jti 使用 UUID4 hex 字符串
- X-Request-Id 使用 UUID4

### 7.4 软删除

- sessions 表使用 status 字段标记软删除（status=0），不物理删除数据
- users 表使用 status 字段标记启用/禁用
- 其他表（documents, messages, handoff_tickets）按需物理删除

### 7.5 安全设计

- 密码使用 bcrypt（rounds=12）哈希后存储，永不明文保存
- SQL 查询使用参数化（%s 占位符），防止 SQL 注入
- 使用显式列名查询，不使用 `SELECT *`，防止敏感字段泄露
- 生产环境需限制数据库仅 127.0.0.1 访问
