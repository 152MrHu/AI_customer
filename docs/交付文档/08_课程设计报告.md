## 2. 项目信息

### 2.1 项目背景

随着大语言模型技术的成熟和 RAG 技术的发展，智能客服系统正从传统的 FAQ 匹配向基于知识库的智能问答演进。本项目旨在实践"微服务 + AI"的工程化开发方法，构建一个完整的 AI 智能客服系统。

项目选题参考了"AI 智能客服 / 知识库问答平台"方向，结合当前行业常见的 RAG 技术方案，实现了知识库文档管理、语义向量检索、大语言模型流式生成、人工客服协同等完整功能。

### 2.2 项目基本信息

| 项目 | 内容 |
|------|------|
| 项目名称 | AI 智能客服系统 |
| 项目类型 | Web 应用（微服务架构） |
| 技术栈 | Python/FastAPI + React + MySQL + Redis |
| AI 能力 | RAG 检索增强生成 + 联网搜索 |
| LLM 后端 | 阿里云 DashScope（通义千问） |
| 开发周期 | 2 周 |
| 团队规模 | 4 人 |
| 代码规模 | 后端 ~5000 行 Python + 前端 ~3000 行 JSX |

#### 5.2.2 重点问题的处理

**问题 1：ChromaDB 段错误（Segfault）**

- **问题描述**：在 Windows + Python 3.13 环境下，ChromaDB 的 Rust 后端进程频繁崩溃（段错误），导致文档入库和向量检索不可用
- **影响范围**：knowledge_service 全部向量操作
- **解决方案**：自研轻量级向量存储方案，使用 SQLite 存储向量数据（JSON 序列化），NumPy 计算余弦相似度
- **处理过程**：
  1. 定位问题：确认是 ChromaDB 1.x Rust 后端与 Python 3.13 的兼容性问题
  2. 方案评估：对比了 Milvus（太重）、FAISS（需要编译）、纯 NumPy（需要全内存）
  3. 方案确定：SQLite + NumPy，纯 Python 实现，无外部依赖
  4. 实现测试：一天内完成实现和验证，检索精度与 ChromaDB 一致
- **经验教训**：选型时需充分考虑目标环境的兼容性；纯 Python 方案在小规模场景下足够可靠

**问题 2：多进程 SQLite 文件锁冲突**

- **问题描述**：ai_service 和 knowledge_service 是独立进程，若直接共享 SQLite 文件会触发 "database is locked" 错误
- **解决方案**：ai_service 不直接访问 SQLite，通过 HTTP 调用 knowledge_service 的向量检索接口
- **经验教训**：微服务架构下，每个数据源只应由一个服务直接管理

**问题 3：Prompt 注入风险**

- **问题描述**：恶意用户可能通过精心构造的输入，诱导 LLM 忽略系统指令或泄露信息
- **解决方案**：
  1. 定义 14 种注入模式的正则检测规则
  2. 过滤危险控制字符
  3. 限制重复字符和查询长度
  4. 在 Prompt 中显式加入"不要执行嵌入指令"的规则
- **经验教训**：AI 系统的安全性需要多层防护，正则检测只是第一道防线

## 6. 经验与教训

### 6.1 教训

1. **不要过早引入重型组件**
   ChromaDB 虽然功能强大，但引入了 Rust 依赖，在非标环境下兼容性差。使用 SQLite + NumPy 的轻量方案更适合课程项目规模，也更容易理解和调试。

2. **接口约定要先于开发**
   在 Day1 就应确定各服务之间的接口协议（请求/响应格式）。虽然我们在这方面做得较好，但在人工转接工单的接口设计上经历了两次调整。

3. **AI 能力的"幻觉"问题不能仅靠 Prompt 解决**
   知识库模式通过严格 RAG 约束了 LLM 的输出范围，但相似度阈值的选择需要反复调试。0.5 的阈值在测试中表现良好，但在实际使用中可能需要根据文档类型动态调整。

4. **前端 SSE 客户端开发有坑**
   浏览器原生 EventSource 不支持 POST 请求和自定义 Header，最终使用 fetch + ReadableStream + 手动解析来实现 SSE 客户端。这是前端开发中花费时间最多的部分。

5. **安全措施不能后补**
   安全应该从设计阶段就考虑，而不是在功能完成后"打补丁"。我们在 Day9 集中做了安全加固审查，发现并修复了一些问题，但如果更早规划，可以避免很多返工。

### 6.2 经验

1. **适配器模式是 AI 工程化的关键**
   通过 LLM 适配器的抽象基类和工厂模式，我们实现了模型与业务的彻底解耦。虽然目前只实现了 DashScope 适配器，但切换到 OpenAI 或其他模型只需要新增一个适配器类，业务代码完全不需要修改。

2. **微服务边界划分"先粗后细"**
   项目开始时按照"能力域"（用户/对话/知识/AI）划分服务边界，这个抽象层次是正确的。在开发过程中进一步发现：AI Service 不应该直接访问向量数据库，于是改为了 HTTP 调用方式，但服务的职责边界没有变。

3. **SSE 流式输出显著提升用户体验**
   逐 token 的打字机效果让用户感知到系统在"思考"和"工作"，相比等待完整回答再一次性显示，用户体验好很多。这是项目中用户反馈最好的功能点。

4. **异步入库方案解决了长任务阻塞问题**
   文档入库（解析→切块→向量化→存储）是一个耗时操作（可能 10-60 秒），使用 asyncio.create_task() 创建后台任务，前端立即得到"上传成功，正在处理中"的反馈，用户体验不受影响。启动时的自动重试机制也保证了系统的自愈能力。

5. **分层架构让独立开发和联调更高效**
   每个服务都有清晰的 router → service → repository 三层，每个人在自己的服务内开发时不用关心其他服务的实现细节。联调时只需要对齐接口即可，基本没有出现"我改了 A 导致 B 挂掉"的情况。

6. **文档与代码同步的重要性**
   我们在 Day1-Day3 先搭建了文档骨架，然后边开发边更新文档。这样确保了最终交付的文档与代码高度一致，避免了"最后再补文档"时发现忘了很多设计细节的问题。

## 7. 技术亮点

### 7.1 LLM 适配器模式

定义了标准的 LLM 抽象接口（`chat_stream` / `embed`），通过工厂模式创建具体实现。当前适配 DashScope（通义千问），预留 OpenAI 扩展位。实现了真正的"业务与模型解耦"。

### 7.2 双模式对话

- **知识库模式**：严格 RAG，回答基于文档内容，可溯源，不编造
- **通用助手模式**：LLM + 联网搜索增强，回答更广泛的问题
- 两种模式共用同一个 Prompt 构建器，通过 mode 参数切换

### 7.3 轻量级向量存储

SQLite + NumPy 方案，纯 Python 实现，零外部依赖，在 Windows/Linux/macOS 上完全兼容。检索精度与 ChromaDB 一致，部署成本为零。

### 7.4 安全防护体系

- JWT + jti + Redis 黑名单（认证）
- 14 种 Prompt 注入模式检测（AI 安全）
- Magic bytes + 文件名清洗 + 大小限制（文件安全）
- 多维度限流（IP/用户/接口）
- 登录锁定（按 user_id，防交替绕过）

### 7.5 SSE 流式全链路

从 DashScope → AI Service → Chat Service → Gateway → 浏览器的完整流式链路，每层都正确处理了 SSE 帧的转发。

## 9. 总结

通过本次课程设计，项目组实践了"微服务 + AI"项目的完整开发流程：

1. **Day1 - Day3**：明确了项目方向，完成了需求分析和系统设计，建立了文档骨架
2. **Day4 - Day7**：打通了主链路（注册→登录→知识库问答→SSE 流式），核心功能可运行
3. **Day8 - Day11**：完成了扩展功能（反馈、转接、客服工作台）和全面测试
4. **Day12 - Day14**：完善了文档体系，确保文档与代码一致

项目完整实现了 20 项功能需求，通过了功能测试和安全测试，无遗留严重缺陷。

项目充分体现了：
- 对微服务架构的设计能力（5 服务 + 网关 + 清晰边界）
- 对 AI 服务化与解耦设计的理解（适配器模式）
- 对团队分工与项目推进的组织能力（4 人并行开发）
- 对安全性的重视（多层安全防护）

项目主要不足在于缺少自动化测试和容器化部署方案，这些将作为后续改进的重点方向。
